## Descriptive Title

Access to system-level stored payment credentials for user-authorized transactions

## Request Type

Interoperability Request

## Feature Name

iOS System-Level Stored Payment Credentials Capability (device-level payment credential
reuse mechanism)

## Reason for Request

Onside is an alternative app marketplace for iOS operating within the EU under the DMA.
iOS provides a system-level capability whereby users can store payment credentials on their
device and use them across system-integrated services without re-entering payment details.
Apple’s own services, including the App Store, rely on this capability: users can complete
transactions using previously configured payment methods without additional data entry.
By contrast, alternative marketplaces such as Onside cannot access this system-level
capability. Users are required to re-enter payment details, even though the same credentials
have already been provided and securely stored at the system level.
This results in duplicate payment setup for the same user on the same device, creating a
material UX disadvantage compared to the App Store.
The relevant functionality is therefore the ability to initiate user-authorized transactions using
system-stored payment credentials without requiring re-entry of payment information.
This functionality is clearly available to Apple’s own services and constitutes a system-level
capability controlled by iOS.
Under Article 6(7) DMA, Apple is required to provide access to the same operating system
features and capabilities that are available to its own services.
This request does not seek access to raw card data or unrestricted financial information.
It seeks access to a scoped payment credential mechanism, which may be implemented using:
- tokenized payment credentials
- user-authorized transaction flows
- standard authentication mechanisms (e.g., 3D Secure, device authentication)
This would allow alternative marketplaces to achieve equivalent transaction UX without
expanding access to sensitive financial data beyond what is already used within the operating
system.
Apple Pay is not an effective alternative. It is a separate payment product with limited coverage
and requires separate setup. It does not provide access to the full set of user-configured
payment methods used by the App Store and therefore does not enable equivalent
functionality.
Without access to system-level payment credentials, Onside faces a structural disadvantage in
transaction completion and monetization, driven solely by Apple’s exclusive integration of this
capability into its own services.

## Product Description

Onside — an alternative app marketplace for iOS that enables developers to distribute their
apps outside the App Store within the EU.

## How the Product Will Use the Feature

Onside will use the iOS system-level payment credential capability to provide a seamless and
secure transaction experience equivalent to the App Store, while maintaining user authorization
and control.
User-authorized transactions using system-stored credentials. When a user initiates a purchase
in Onside, the marketplace will present the payment methods already configured at the system
level (e.g., “Visa ending in 4242”). The user selects a preferred method and confirms the
transaction. Onside will then initiate the payment using system-provided, tokenized credentials,
with appropriate authentication where required (e.g., device authentication or 3D Secure).
No re-entry of payment details. Users will not be required to re-enter card or payment
information that has already been securely stored at the system level. This eliminates duplicate
payment setup for the same user on the same device and aligns the transaction flow with the
App Store experience.
Support for subscriptions and in-app purchases. Onside will use system-level payment
credentials to support recurring transactions, including subscriptions and in-app purchases,
with appropriate user authorization and lifecycle management (e.g., renewals, cancellations,
refunds).
Transaction processing with clear boundaries. Onside will initiate and manage transactions
within its own marketplace environment while relying on system-provided payment credentials
and standard authentication mechanisms. At no point will raw payment details be exposed to
or handled directly by Onside.
User control and transparency. Users will have clear visibility into selected payment methods
and transaction confirmations. Standard controls (e.g., authentication prompts, cancellation,
refund flows) will be available and consistent with existing iOS patterns.

## Other Frameworks Evaluated

We have evaluated the following alternatives:
1. Asking users to enter payment information in Onside: This is what Onside and all alternative
marketplaces are currently forced to do. Users must enter their card details from scratch, even
though the same payment methods are already stored in their Apple ID. This creates the exact
friction problem we are addressing: the same user, on the same device, must duplicate their
payment setup simply because they chose an alternative marketplace.
2. Apple Pay integration: Apple Pay is not equivalent to the system-level payment credential
capability used by the App Store. It is a separate payment product with different scope and
behavior:
- Apple Pay requires its own setup flow and is not automatically enabled for all users who have
configured payment methods at the system level
- It supports only a subset of payment methods (primarily cards provisioned to Apple Pay),
whereas the App Store supports a broader set (e.g., cards, PayPal, carrier billing, and other
region-specific methods)
- It operates through a distinct interaction model (sheet-based authorization) and is not the
mechanism used by the App Store for marketplace transactions
- It does not provide access to the full set of user-configured payment credentials or enable
equivalent transaction flows for app marketplace purchases
As a result, Apple Pay does not allow alternative marketplaces to replicate the functionality or
user experience of the App Store.
None of these alternatives eliminate the UX friction created by iOS's exclusive integration of
system-level stored payment methods with App Store. Only direct access to payment tokens
for the user's stored payment methods can provide interoperability.

## Additional Information

Privacy Safeguards:
Onside’s use of the system-level payment credential capability is designed to minimize data
exposure and align with platform security standards, PCI DSS requirements, and EU data
protection regulations.
Tokenized and minimal data access
Onside will access only tokenized payment credentials and limited metadata necessary for
transaction presentation and processing (e.g., payment method type, card network, last four
digits). Full card numbers, CVV codes, and underlying financial data will not be accessible.
No extraction or storage of sensitive payment data
Payment credentials will remain within system-controlled secure environments. Onside will not
extract, store, or export sensitive payment data beyond what is strictly necessary to initiate and
complete user-authorized transactions.
Purpose limitation
All access to payment credentials will be strictly limited to:
- processing user-authorized transactions
- managing subscriptions and renewals
- handling refunds and payment-related support
No payment data will be used for profiling, advertising, or shared with third parties beyond
what is required for transaction execution (e.g., payment processors).
User authorization and control
Every transaction will require explicit user selection of a payment method and confirmation.
Users retain full control over their payment methods via system settings and can manage, add,
or remove them independently of Onside.
Security and compliance
Access to the payment capability will be governed by iOS entitlements, secure APIs, and audit
logging. Onside will comply with applicable PCI DSS requirements and GDPR obligations,
including data minimization, purpose limitation, and user rights (access, deletion, portability
where applicable).
Technical Details:
- iOS already provides system-level payment method access to App Store and other Apple
services. The capability exists and is operational.
- Apple can expose a payment method API to authorized alternative marketplaces that returns
payment tokens and sufficient details (card network, last four digits, tokenized credentials) for
independent payment processing.
- Implementation would follow Apple's existing pattern for system API access (entitlements,
capability tokens, standard security review).
Success Criteria:
- Onside users can complete purchases using pre-configured Apple ID payment methods
without re-entering payment information (matching App Store UX).
- Onside can independently process payments using tokenized credentials, including 3D
Secure flows where required.
Users see the same payment methods in Onside that they have configured in their Apple ID.
