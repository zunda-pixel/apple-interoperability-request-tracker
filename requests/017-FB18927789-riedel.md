## Descriptive Title

Allow 3rd party apps to use Screen Time’s DeviceActivityReport API in Widgets just like Apple’s
Screen Time Widget

## Request Type

Interoperability Request

## Feature Name

Screen Time Widget

## Reason for Request

Hello, my name is Frederik Riedel and I’m the developer behind an app called “one sec”.
Background information / context of my app:
With one sec, we help millions of users around the world to save more than 1400+ years (!)
from scrolling social media *every single week*.
We help grown-up adult users to limit their own screen time (NOT adolescents who are
managed by their parents).
one sec is using Shortcuts automations in order to react to users opening target apps: an
intervention (e.g. breathing exercise) is injected before social media apps can be used – so the
user can re-think if they really want to continue to the target app – or if it was just a muscle
memory reflex.
Link to the app: https://apps.apple.com/app/apple-store/id1532875441
We are also conducting many research projects in one sec, for example with the German Max-
Planck Institute, Stanford University, or Oxford University in the UK, e.g.: https://www.pnas.org/
doi/10.1073/pnas.2213114120
We were able to show that one sec reduces social media usage by 57% on average.
One big feature set of one sec are statistics, and using them to educate / nudge users for
healthier smartphone habits.
For that we make use of the DeviceActivityReport API, which is a sandboxed environment
which lets 3rd party render iOS Screen Time data without actually accessing the data.
This is done in a very privacy conserving way which is actually a great idea (although it is not
perfect and requires some bug fixes from Apple).
We would love to make use of this framework inside of Widgets, however that does not work.
DeviceActivityReport rendered inside of Widgets remain blank.
Apple however can make use of Screen Time data within widgets (see attached screenshot)
and we would love to offer similar functionalities for our users.

August 20, 2025, from Developer to Apple:
Dear Apple Developer Relations Team,
Thank you for your response and for reviewing my interoperability request regarding the use of
the DeviceActivityReport API in widgets.
I understand and appreciate your position that the API is technically available and that the
behavior I’m seeing may be classified as a bug. At the same time, I’d like to offer some
additional context that might help clarify why I believe this is an interoperability issue — and
why addressing it would be of real value to developers and users alike.
From a user perspective, the distinction between bug vs. feature vs. interoperability limitation
doesn’t matter much — what matters is that helpful features are available in a consistent and
privacy-preserving way across the ecosystem.
Our app one sec supports millions of users in reducing social media usage and building
healthier habits. We’ve partnered with research institutions such as the Max Planck Institute,
Oxford, and Stanford to explore the impact of our interventions, and we’ve found that one sec
can reduce social media use by over 57% — saving more than 1400 years of attention every
single week.
One important part of this is showing meaningful and privacy-focused usage statistics — and
widgets are the perfect place for this. However, we’ve found that using the
DeviceActivityReport API in a widget simply renders a blank view.
We’ve confirmed that this behavior has been raised by other developers before — including a
thread on Apple’s own developer forums from January 2023, where an Apple Frameworks
Engineer responded:
“Unfortunately, displaying a DeviceActivityReport in a widget is not currently supported. Please
file an enhancement request using Feedback Assistant.”
Source: https://developer.apple.com/forums/thread/724041?answerId=743998022#743998022
More than two years later, the limitation remains — which is why I submitted this as an
interoperability request. Apple’s own Screen Time widget can provide similar functionality, but
this path is not available to third-party developers, despite using the same core technology.
This means our apps cannot provide users with comparable tools for monitoring their usage —
even though the API itself exists and works in-app.
To me, this feels like a textbook case of “ineffective interoperability”: the system feature is
technically present but not practically usable in third-party contexts like widgets. And as
pointed out in your Frameworks Engineer’s response, this is more than a simple bug — it’s a
known design limitation.
I would love to find a solution here. Whether it’s reclassifying the widget restriction as an
interoperability concern, enabling it through entitlement, or revisiting the sandbox behavior —
I’m open to any path forward that enables this use case.
If helpful, I’d be happy to share technical details, mockups, or concrete user feedback to
support the discussion.
Many thanks again for your time,
and I look forward to hearing your thoughts.
Warm regards,
Frederik Riedel
Developer of one sec

October 15, 2025 from Apple to Developer:
Thank you for the additional information. Screen Time is a part of Settings and therefore a part
of iOS itself.
As you know from prior requests which we have progressed, certain Screen Time features that
are accessed and used by Apple services may be in-scope under Article 6(7) DMA. In this
instance, Apple’s display of Screen Time data from Settings in a system widget is a
functionality of the OS. As no Apple service is accessing a Screen Time feature in this scenario,
this falls outside the scope of Article 6(7) DMA.
This concludes our review under Article 6(7) DMA.
