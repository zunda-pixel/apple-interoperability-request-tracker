## Descriptive Title

Access to system-level automatic authentication capability for device-authenticated users

## Request Type

Interoperability Request

## Feature Name

iOS System-Level Automatic Authentication Capability (device-level credential reuse
mechanism)

## Reason for Request

Onside is an alternative app marketplace for iOS operating within the EU under the DMA.
iOS provides a system-level authentication mechanism whereby users authenticated at the
device level (via Apple ID) can access certain system-integrated services without additional
credential entry.
The App Store relies on this capability: when users open the App Store, they are automatically
authenticated based on their device-level identity, without any additional login or user
interaction.
By contrast, alternative marketplaces such as Onside cannot access this system-level
authentication capability. Users are required to complete a separate authentication flow,
including entering credentials and creating an account.
This results in duplicate authentication for the same user on the same device, creating a
material UX disadvantage compared to the App Store.
The relevant functionality is therefore the ability to authenticate a user based on device-level
credentials without requiring additional user interaction.
This functionality is clearly available to Apple’s own App Store and constitutes a system-level
capability controlled by iOS.
This does not create new access to user identity or device data; it only enables reuse of an
authentication state that is already established at the operating system level.
Under Article 6(7) DMA, Apple is required to provide access to the same operating system
features and capabilities that are available to its own services.
This request does not seek unrestricted access to Apple ID or sensitive personal data.
It seeks access to a scoped system-level authentication mechanism, which may be
implemented using:
- a pseudonymous, device-linked user identifier
- optional access to basic profile attributes subject to user consent
This would allow alternative marketplaces to achieve equivalent authentication UX without
expanding access to user identity beyond what is already available on the device.
Sign in with Apple is not an effective alternative, as it requires explicit user interaction and does
not provide automatic authentication. It operates at the application level rather than as a
system-level capability.
Without access to system-level authentication, Onside faces a structural disadvantage in user
onboarding and account management, driven solely by Apple’s exclusive integration of this
capability into its own services.

## Product Description

Onside — an alternative app marketplace for iOS that enables developers to distribute their
apps outside the App Store within the EU.

## How the Product Will Use the Feature

Onside will use the iOS system-level authentication capability to provide a seamless, low-
friction user experience equivalent to the App Store, while maintaining appropriate user
transparency and control.
Automatic authentication based on device-level credentials. When a user opens Onside, the
app will automatically authenticate the user based on their existing device-level authentication
state, without requiring additional credential entry. This avoids duplicate authentication for the
same user on the same device and aligns with the experience provided by the App Store.
Account initialization and continuity. Onside will use a system-provided, stable, pseudonymous
user identifier to initialize and maintain a user account within the marketplace. This allows
continuity across sessions, app reinstalls, and device usage without requiring manual account
creation.
User profile and account management (optional, user-controlled). Onside may display basic
account information and provide account management functionality within the marketplace.
Any access to user-identifying attributes (such as name or email) will be optional and subject to
explicit user consent, consistent with iOS permission patterns.
Purchase association and lifecycle management. Onside will use the system-provided identifier
to associate purchases, subscriptions, and entitlements with the authenticated user. This
enables core marketplace functionality such as purchase history, refunds, subscription
management, and license validation — equivalent to the capabilities available to the App Store.
At all times, authentication will be based on the user’s existing device-level authentication
state, with clear user controls (e.g., sign-out, reset, or reauthorization) available within the app.

## Other Frameworks Evaluated

We have evaluated the following alternatives:
1. Sign in with Apple (ASA): ASA is a third-party authentication method presented alongside
other OAuth providers. It is not equivalent to system-level identity resolution. Users must
explicitly initiate the Sign in with Apple flow, creating friction compared to App Store's
automatic authentication. Additionally, ASA is designed for cross-platform use and does not
provide the device-backed unique identifier that system-level API access would provide.
Further, Apple controls ASA and has previously limited its availability to certain use cases.
2. Building proprietary Onside authentication: Onside could maintain its own user identity
system requiring email and password. This is what Onside currently does and what all
alternative marketplaces are forced to do. However, this perpetuates the UX gap: App Store
users authenticate once during device setup; Onside users must authenticate again when first
opening the marketplace. This is a feature-driven disadvantage, not a technical limitation.
3. Device-level MDM or enterprise enrollment: Mobile Device Management is designed for
organizational control, not user-level marketplace authorization. It is not a viable path for
consumer-facing distribution and would require administrative overhead incompatible with
personal device use.
4. Requesting Sign in with Apple enhancement: Apple could theoretically enhance Sign in with
Apple to provide automatic authorization (without user interaction) and device-backed
identifiers. However, Apple controls this feature and has shown no inclination to extend it in this
direction. Additionally, even enhanced ASA would remain under Apple's control and subject to
Apple's policies, creating ongoing interoperability risk.
None of these alternatives provide effective interoperability. Only direct access to the iOS
System-Level User Profile API can eliminate the structural authentication friction gap between
App Store and Onside.

## Additional Information

Privacy Safeguards:
Onside’s use of the system-level authentication capability is designed to minimize data access
and align with both iOS platform principles and EU data protection requirements.
Minimal and scoped data access
By default, Onside will rely on a system-provided, pseudonymous, device-linked user identifier
for authentication and account management. This identifier will not directly reveal the user’s
identity and will be scoped to Onside’s marketplace context.
No unrestricted access to Apple ID or sensitive identity data
This request does not require or seek direct access to Apple ID credentials or unrestricted
personal data. Any user-identifying attributes (such as name or email address) will only be
accessed if explicitly provided by the user through a consent-based mechanism.
Purpose limitation
All data accessed through the system-level authentication mechanism will be used strictly for:
- user authentication
- account management
- purchase tracking and history
- subscription and entitlement management
- refunds and customer support
No data will be used for advertising, profiling, or shared with third parties beyond what is
necessary to operate the marketplace.
User transparency and control
Users will be able to:
- understand when authentication occurs
- access and manage their account within Onside
- revoke or reset their association with the marketplace
- request deletion of their data in accordance with GDPR
Security and compliance
Access to the authentication capability will be governed by standard iOS entitlement
mechanisms, audit logging, and security review processes. Onside will comply with all
applicable GDPR obligations, including data minimization, purpose limitation, and data subject
rights.
Technical Details:
- iOS already provides system-level user profile access to App Store via internal integration.
The capability exists and is operational.
- Apple can expose a scoped user profile API to authorized alternative marketplaces (those
attesting to DMA compliance) that returns a stable, pseudonymous identifier by default, with
optional access to user attributes (such as name or email) only upon explicit user consent
- Implementation would follow Apple's existing pattern for system API access (entitlements,
capability tokens, standard security review).
Success Criteria:
- Onside achieves automatic user authentication without requiring users to enter email or
password on first app open (matching App Store UX).
- Onside can display the user's full name and email within the marketplace and provide
account settings — matching App Store's account management experience.
- Onside can reliably associate purchases with the authenticated user, enabling accurate
purchase history, subscription management, and refund processing.
- No user privacy incidents or unexpected data exposure resulting from profile access.
