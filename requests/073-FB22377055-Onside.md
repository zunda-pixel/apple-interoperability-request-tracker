## Descriptive Title

Access to system-level secure communication functionalities between alternative marketplaces
and their distributed apps

## Request Type

Interoperability Request

## Feature Name

iOS System-Level Entitlement and Transaction Delivery Functionalities (store-to-app
communication functionalities)

## Reason for Request

Onside is an alternative app marketplace for iOS operating within the EU under the DMA.
iOS provides a system-level framework that enables secure communication between the App
Store and the apps it distributes. This framework (StoreKit) is a software feature allowing apps
to rely on the operating system for transaction execution, entitlement delivery, and purchase
verification.
When a user purchases an app or in-app product via the App Store, the operating system
executes the transaction using system-stored payment credentials and delivers the
corresponding transaction results and entitlement data directly to the app. Apps can query this
system-level framework to determine the user’s purchase status and unlock functionalities
accordingly, without requiring additional authentication, custom backend infrastructure, or per-
app integration.
Importantly, this capability does not expose user credentials or payment data to apps. The
operating system acts as a trusted intermediary that enables apps to rely on a system-
managed transaction context without requiring separate user authentication, executes user-
authorized transactions, and provides apps with signed transaction and entitlement
information.
By contrast, alternative marketplaces such as Onside do not have access to any equivalent
system-level capability. Apps distributed via Onside cannot rely on the operating system to
manage transactions or deliver entitlements and must instead implement their own
authentication systems, payment infrastructure, and verification logic.
This results in a structurally degraded experience:
- users must sign in separately within each app
- payment methods must be reconfigured or re-entered
- purchases cannot be seamlessly restored or verified
- developers must build and maintain infrastructure that is not required when using the App
Store
The relevant functionality is therefore the ability for the operating system to act as a secure
intermediary between a marketplace and the apps it distributes, enabling:
- execution of user-authorized transactions using system-stored payment credentials
- delivery of transaction results and entitlement data to apps
- verification and restoration of purchases through a system-level interface
This functionality is clearly available to Apple’s own App Store and constitutes a system-level
capability controlled by iOS.
Under Article 6(7) DMA, Apple is required to provide access to the same operating system
features and capabilities that are available to its own services.
This request does not seek access to user credentials or raw payment data.
It seeks access to system-level communication and transaction functionalities, equivalent in
function to StoreKit, that enables alternative marketplaces and their distributed apps to rely on
the operating system for transaction execution and entitlement management.
Without access to such system-level functionalities, apps distributed outside the App Store are
structurally disadvantaged, as they must recreate functionalities already provided by the
operating system to Apple’s own services, resulting in fragmented user experience and
increased developer burden.

## Product Description

Onside — an alternative app marketplace for iOS that enables developers to distribute their
apps outside the App Store within the EU.

## How the Product Will Use the Feature

Onside will use the system-level store-to-app communication framework to enable secure
delivery of transaction results and entitlement information from the marketplace to the apps it
distributes, without exposing user credentials or payment data.
Transaction and entitlement delivery. When a user completes a purchase through Onside, the
operating system will deliver the corresponding transaction result and entitlement data to the
app when it is opened or queried. This allows the app to unlock purchased features without
requiring additional user interaction.
Seamless in-app purchases. When a user initiates an in-app purchase within an Onside-
distributed app, the operating system will execute the transaction through a user-authorized
flow and deliver the resulting entitlement status to the app. The app does not need to handle
payment credentials or implement custom transaction logic.
Purchase restoration. Apps will be able to query the system-level framework to restore
previously purchased entitlements across devices or reinstalls, consistent with existing iOS
patterns.
Standardized system interface for developers. Developers will be able to rely on a system-level
interface for entitlement verification and transaction handling, without requiring marketplace-
specific SDKs or custom backend infrastructure for each distribution channel.

## Other Frameworks Evaluated

We have evaluated the following alternatives:
1. Ask the user to sign in separately in every installed app: Each app distributed via Onside
could require the user to sign in to the same Onside account to access stored payment
methods. This creates significant friction: the user must enter credentials in every app, rather
than having a seamless one-tap purchase experience. By contrast, apps installed from the App
Store require no separate sign-in — StoreKit handles everything at the system level.
2. Redirect the user through Onside app via Universal Links: When an app needs user
credentials, it could open the Onside app, which would authenticate the user and then redirect
back to the installed app, passing session or authentication data through a Universal Link. This
is not a viable solution: it is insecure (credentials are passed through URL parameters), creates
a disruptive user journey with multiple app switches and system confirmation alerts, and is
unreliable across different iOS versions and states.
3. Ask the user to add payment methods from scratch in every installed app: Each app could
implement its own payment setup, requiring the user to enter card details independently in
every app they use. This is the worst user experience — complete duplication of effort for each
app — and the opposite of the seamless payment experience that StoreKit provides to App
Store-distributed apps.
None of these alternatives provide equivalent interoperability to StoreKit. Only a system-level
communication channel between the marketplace and its distributed apps can eliminate the
friction gap.

## Additional Information

Privacy Safeguards:
The system-level communication framework will be designed to minimize data exposure and
align with existing iOS security and privacy principles.
Scoped entitlement data only. Apps will receive only the transaction and entitlement
information necessary to determine purchase status (e.g., product identifiers, purchase state,
and validity). No unrelated personal data will be shared.
No credential or payment data exposure. The framework will not expose user credentials,
payment methods, or financial data to apps or to Onside. All sensitive operations remain within
system-controlled environments.
No cross-app data sharing. Each app will receive only information related to its own
transactions and entitlements. No cross-app analytics or behavioral data will be available.
Secure and verifiable communication. All transaction data will be cryptographically signed and
verifiable by apps, preventing tampering or spoofing, consistent with StoreKit receipt validation
mechanisms.
User privacy and compliance. All processing will comply with iOS entitlement controls and
GDPR requirements, including data minimization and purpose limitation.
Technical Details:
- iOS StoreKit already implements secure store-to-app communication for App Store. The
technical patterns and security infrastructure are proven and operational.
- Apple can extend the framework (or create a parallel equivalent) to support authorized
alternative marketplaces.
- Implementation would use the same receipt signing, cryptographic validation, and
entitlement-based access control as current StoreKit.
- Receipt format can remain compatible with existing app expectations, reducing developer re-
implementation burden.
Success Criteria:
- Apps distributed via Onside can deliver in-app purchase functionality with a seamless user
experience equivalent to StoreKit.
- In-app purchase completion rates for Onside-distributed apps are equivalent to App Store
completion rates, indicating feature parity.
- Developer satisfaction: developers report that integrating in-app purchases for Onside-
distributed apps requires equivalent effort to App Store integration (no additional per-store
SDKs).
- Subscription retention and revenue metrics for Onside-distributed apps meet or approach
industry benchmarks for subscription apps.
- No security incidents or purchase validation failures attributable to the store-to-app
framework.
