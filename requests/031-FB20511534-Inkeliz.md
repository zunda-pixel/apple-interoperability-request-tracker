## Descriptive Title

Push-Notification over Internet without APNs

## Request Type

Interoperability Request

## Feature Name

Push Notification

## Reason for Request

We aim to develop applications that connect directly with IoT devices, which must send push
notifications directly to iOS apps. Currently, this is not possible due to Apple’s exclusive control
of the Apple Push Notification Service (APNs).
APNs relies on system-level features unavailable to third-party developers. While Apple allows
third parties to use APNs, it requires routing messages through a central, trusted server. This
creates an architecture like: iPhone ↔ APNs ↔ App Developer Server ↔ IoT Device
We also intend to build a decentralized chat application where each user can connect to
independent servers. APNs prevents this design because push notifications must be routed
through a single trusted server with a valid APNs key or token. These credentials cannot be
safely shared with third parties, and users cannot receive push notifications directly from
arbitrary servers that bypass APNs.
We choose not to use APNs for privacy and cost reasons. We do not want to process customer
data on our servers, even if encrypted, and we aim to minimize latency by allowing direct
device-to-app communication.
All push notifications on iOS and iPadOS must pass through APNs. This restriction prevents us
from creating an equivalent notification service or maintaining our own secure and reliable
delivery path. It also limits innovation, as every notification must flow through an additional
layer (APNs ↔ Developer Server ↔ IoT Device), adding complexity and reducing reliability.
From a technical perspective, Apple devices maintain a persistent background connection only
with APNs, even when apps are closed or the device is locked. Third-party developers cannot
maintain live connections to their own servers or perform background updates under the same
conditions.
This exclusive capability gives APNs a functional advantage over other services. To ensure fair
interoperability and equal access to iOS system features, we request technical documentation
or APIs that enable equivalent background connectivity and push-notification functionality for
third-party services.

## How the Product Will Use the Feature

Push notifications are essential for chat applications and IoT systems. We request
interoperability that allows devices and servers to send push notifications directly to users
without relying on APNs. This would enable secure, real-time device-to-user communication,
reduce latency, and improve reliability by avoiding mandatory routing through external Apple-
controlled servers.
APNs limits users and developers to a single centralized system. It prevents connection to
multiple independent servers, each responsible for notifications in its own message channel.
Interoperability would support a federated model where independent communication providers
can operate on equal terms and deliver notifications directly to iOS devices.
This approach also strengthens user privacy by allowing data to remain within jurisdictions not
subject to the U.S. CLOUD Act or other foreign data-access laws.

## Other Frameworks Evaluated

Currently, Apple already provides "Local Push Connectivity" (https://developer.apple.com/
documentation/networkextension/local-push-connectivity), but it's limited to local networks.
We need a similar mechanism, but not restricted to local networks and capable of connecting
(and keep connected to) arbitrary servers.
That would replace APNs and makes possible to compete with APNs, allowing developers (and
maybe users) to choose what service wants to use, including their own servers.
In the user-side, if battery life (or privacy) are a concern: Apple could, optionally, keep the
connection alive outside of the device (in that case: Device <-> Apple Relay <-> IoT Device*).
That is similar to features that Apple already provides, like "iCloud Private Relay". That should
be optional and a user choose.

## Additional Information

We have evaluated Apple’s existing frameworks, including UserNotifications, PushKit,
Background App Refresh, and Network Framework for local communication. None of them
enable effective, reliable push notifications outside APNs.
UserNotifications and PushKit both depend entirely on APNs. Background App Refresh is
limited by strict scheduling and power constraints, preventing real-time or near-real-time
updates.
We also tested local network push mechanisms, where a device would receive notifications
directly from another device or a local hub within the same LAN or Wi-Fi network. However,
iOS restricts background sockets and multicast/broadcast traffic when the app is suspended
or closed. As a result, devices cannot reliably receive notifications locally without going through
APNs.
We experimented with MQTT and WebSocket-based solutions, but iOS terminates these
connections once the app enters the background. No available framework allows persistent or
periodic background communication for IoT, decentralized chat, or local network notifications
without depending on APNs.
Furthermore, APNs require every notification to pass through Apple’s servers, which prevents
device-to-device or local delivery. This creates latency, increases dependency, and raises
privacy concerns because all traffic is routed through Apple’s infrastructure, even for local or
internal communication. APNs is not designed for IoT or decentralized networks. It cannot
dynamically address multiple independent servers or manage connections for third-party
notification systems. Each app must register for a token linked to Apple’s backend, and those
tokens cannot be shared or resolved outside Apple’s ecosystem.
APNs is currently using iOS features that no other developer can use.
