## Descriptive Title

Provide same robust ShieldSettings.ActivityCategoryPolicy.all(except: …) API for 3rd party
developers, as being used by Apple’s Screen Time

## Request Type

Interoperability Request

## Feature Name

Screen Time

## Reason for Request

We are requesting Apple’s support in resolving a long-standing technical issue with the
ShieldSettings.ActivityCategoryPolicy.all(except:) API.
This API is intended to allow developers to block access to all apps on a user’s device, except
for a defined set of allowed applications. However, in practice, the exempted apps are still
shielded with a generic “App Restricted” overlay — even though they were explicitly excluded
in the configuration.
This undermines the expected behavior of the API and significantly limits the ability of
developers to offer reliable parental controls and digital wellbeing interventions.
Apple’s own Screen Time implementation can exempt apps from restrictions without such
issues (screenshot attached, iOS Settings → Screen Time → Always Allowed).
Therefore, this appears to be an internal capability not currently available under the same
conditions to third-party developers.
We believe resolving this would not only improve technical consistency, but also allow for more
robust interoperability and innovation in the Screen Time and parental control ecosystem.
A sample project and screen recording demonstrating the issue are attached.

## How the Product Will Use the Feature

Our apps use the ShieldSettings.ActivityCategoryPolicy.all(except:) configuration to block all
apps except those explicitly permitted by the user (e.g. meditation apps, contacts, notes,
phone, maps, Uber).
Due to the issue described, these exempted apps are still visually shielded, which leads to
confusion and a loss of trust in the intervention — despite them technically remaining
accessible.
This is especially important if users want to block all distractions on their phone (Instagram,
YouTube, Netflix, Tinder, Candy Crush, TikTok…) while maintaining access to useful &
important tools such as Maps, Facetime, or Uber.
This inconsistent API behavior limits our ability to deliver effective and trustworthy digital
wellbeing solutions, even though the underlying API appears to intend otherwise.

## Other Frameworks Evaluated

We have explored multiple technical workarounds, including alternative configurations within
the FamilyControls API, however none of them lead to the desired outcome.
Apple’s own implementation of Screen Time does not exhibit this issue — suggesting that it is
technically solvable and already supported internally.
However, no public entitlement or workaround has been offered to date, and the bug has
remained unresolved since the API was introduced in 2020.

## Confidential Treatment

Fully available to other developers

## Additional Information

This inconsistency between capabilities provided to 3rd party developers and Apple’s own
Screen Time technology has existed since the Screen Time API framework has been
introduced in 2020.
The issue has been raised before in this radar: FB15190710 but remains without solution. This
is why we have no choice than to raise this again via an interoperability request – to ensure 3rd
party developers get access to the same reliable API as well.
