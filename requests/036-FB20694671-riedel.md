## Descriptive Title

Provide same consistent & stable ApplicationTokens that Apple uses for their own Screen Time
solution

## Request Type

Interoperability Request

## Feature Name

Screen Time

## Reason for Request

We rely on Apple’s Family Controls framework and Screen Time shield extensions to deliver a
core functionality of our app — helping users make more intentional decisions before opening
time-consuming apps like social media.
However, starting with iOS 17.5 (and continuing in iOS 18 and 26), we have encountered a
serious and inconsistent bug: the ApplicationToken passed into the ShieldActionExtension and
ShieldConfigurationExtension often does not match the token the user previously selected via
the FamilyActivityPicker. This leads to broken logic, confusing shield behavior, and a
significantly degraded user experience.
Since the ApplicationToken is the only mechanism we are given to identify which app is being
shielded, and we are not allowed to map it to a specific bundleIdentifier, this inconsistency
breaks our integration entirely — and there is no workaround available.
Apple’s help is required to resolve this bug and ensure stable and predictable API behavior in
line with the Family Controls framework documentation – similar to how Apple’s own Screen
Time solution is able to block apps reliably.

## How the Product Will Use the Feature

We use FamilyActivityPicker to allow users to select which apps they want to shield (e.g.,
Instagram, TikTok, YouTube). The system provides us with ApplicationTokens corresponding to
these selections.
Our app uses Apple’s ShieldConfigurationExtension to block apps for certain reasons (e.g.
during Focus, during specific times of the day, or on demand) or to restrict usage after a certain
limit has been reached (e.g. after 3 minutes, to avoid doom scrolling).
To ensure the correct logic and display, we depend on the shield extensions receiving the exact
same ApplicationToken that the user selected. Apple’s focus on privacy here is great. But it
breaks the user experience if the API does not behave according to the documentation and
new ApplicationTokens are continuously made up.
Due to the bug, the ApplicationToken passed into the ShieldConfigurationExtension and
ShieldActionExtension often does not match any selected / known token — resulting in shields
being shown for the wrong app, not shown at all, or shown for the wrong reason (e.g. Limit
reached instead of Block during Focus).
This breaks both the logic and user trust.
Apple’s own Screen Time solution seems to have access to a more robust, less obfuscated
version of the framework, which does not cause such token misalignment issues.

## Other Frameworks Evaluated

- Custom VPN-based solutions: We deliberately avoid these due to privacy, user trust, and App
Store compliance concerns.
- Shortcuts automations: We already use this approach for additional interventions, but it
cannot offer the same level of system integration and coverage as the shield APIs. On top of
that, the setup process is much more advanced and technical.
- Building our own shielding system: Not possible within Apple’s sandboxing model — iOS
does not allow us to intercept app launches ourselves.
