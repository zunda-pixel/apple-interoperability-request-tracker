## Descriptive Title

Request for interoperable, network-based messaging client support on Windows

## Feature Name

Messages (iMessage, RCS, SMS< MMS) – multi-device messaging client

## Reason for Request

Windows is currently limited to Bluetooth Message Access Profile (MAP) for messaging
interoperability with iOS. MAP only enables basic, text-only, one-to-one messaging and does
not support modern messaging capabilities such as group chats, images, attachments,
reactions, or richer messaging protocols. As a result, third-party platforms are unable to offer a
high-fidelity messaging experience that interoperates with iOS Messages.
This request is to support extending messaging access beyond MAP so that messaging
experiences on Windows can achieve parity with Apple’s own multi-device messaging model,
including support for group chats, images, attachments, and reactions.
On macOS, Apple provides a first-party Messages client that allows macOS devices to send
and receive iMessage, RCS, MMS, and SMS messages under the same Apple ID as the user’s
iPhone. This client supports rich messaging capabilities and operates without the macOS
device being the primary cellular endpoint.
For users, who rely on an iPhone alongside a Windows PC, the messaging experience today is
fragmented, leaving them effectively locked out of a rich, modern messaging experience
outside of macOS. While users on mac can seamlessly continue conversations across the
devices they actively use, Windows is constrained to inadequate messaging capabilities that
don’t support group chats, media, or rich interactions, creating daily friction in communication
workflows.
Apple already demonstrates a more complete, user-centric model on macOS, where a Mac can
act as a full-fidelity messaging client without being the primary cellular device, using the same
Apple ID as the user’s iPhone.  Given that Apple already enables rich, multi-device messaging
on macOS under this model, similar functionality could be made available to other platforms,
such as Windows, through documented APIs, entitlements, or a supported client model.
Apple’s help is required because the current limitations are driven by the platform policy and
the lack of exposed interfaces, rather than technical constraints that third parties can address
independently.

## Product Description

Windows, a general-purpose personal computing platform used globally for productivity,
communication, and multi-tasking across consumer and enterprise scenarios.

## How the Product Will Use the Feature

Windows would use interoperable messaging support to help iPhone users stay connected to
their conversations while working on a Windows PC, without forcing them to switch devices or
disrupt their workflow. From the customer’s perspective, this enables their PC to act as a
trusted secondary messaging client – similar in principle to how macOS participates in the
Apple messaging ecosystem – so messaging feels continuous and consistent across the
devices they actively use.
Specifically, this would allow users to:
•
Send and receive messages from their Windows PC using the same Apple ID as their
iPhone
•
Participate in existing conversations, including group chats
•
Access rich messaging features such as images, attachments, reactions, and modern
messaging protocols (e.g. iMessage and RCS), rather than being limited to basic text.
The intent is not to replace iOS Messages, but to allow users who choose Windows PCs to
remain fully connected to their messaging conversations while working on their PC.

## Other Frameworks Evaluated

Yes. We currently rely on Bluetooth Message Access Profile (MAP), which is the only available
mechanism for third-party messaging interoperability with iOS messages today.  However,
MAP, as implemented currently, provides only limited functionality:
•
Text-only messages
•
1-to-1 conversations
•
No support for images, attachments, reactions or group chats
•
No support for modern messaging experiences
•
No support for maintaining message history
These limitations prevent third party devices and platforms from delivering a messaging
experience that meets user expectations or matches the capabilities Apple provides on
macOS. The messaging-app entitlement enables third-party messaging services a means to
replace the default experience but does not enable third party devices and platforms to
interoperate with the existing messaging experience.

Messaging interoperability is a core user expectation in cross-device workflows. Users who
choose Windows PCs alongside iPhone experience a significant functional gap compared to
user who use iPhones with macOS, despite using the same Apple ID and participating in the
same messaging ecosystem.
The existence of a network-based, Apple ID-backed messaging client on macOS demonstrates
that this capability is feasible without compromising device security or user privacy. We believe
there is an opportunity to work collaboratively with Apple on an approach that enables
meaningful interoperability while respecting Apple’s platform principles and security
requirements.
We would welcome engagement on potential models, including supported APIs, entitlements,
or alternative mechanisms that would allow non-Apple platforms to participate as trusted
messaging clients under clearly defined constraints.
