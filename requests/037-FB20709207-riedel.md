## Descriptive Title

Open up API to render collections of ApplicationToken / ActivityCategoryToken /
WebDomainToken as localized user-facing strings / labels

## Request Type

Interoperability Request

## Feature Name

Screen Time

## Reason for Request

Currently, third-party developers have no way to render multiple ApplicationToken and/or
ActivityCategoryToken and/or WebDomainToken objects together in a single localized, user-
facing string — as Apple’s own Screen Time UI does when summarizing selections of blocked
apps, app categories, and web domains.
The only available APIs Label(applicationToken), Label(webDomainToken), or
Label(activityCategoryToken) work for individual items only.
For example, when a user selects multiple apps, categories, and websites to be limited, there
is no API to produce a localized string such as “Instagram, TikTok and Games”
This limitation forces third-party developers to provide vague or less user-friendly summaries
like “2 apps and 1 category selected,” which results in a worse user experience than Apple’s
own Screen Time interfaces.
To ensure transparency and clarity for users, we require the same system-level localization and
presentation logic that Apple uses internally to render collections of tokens as readable names.

## How the Product Will Use the Feature

When users configure one sec, they select which apps, categories, or websites they want to
intercept or limit.
We would use the requested API to display a user-friendly preview summarizing their selection
— for example:
Selected: Instagram, TikTok, and Games
This improves clarity and trust by showing users exactly what will be affected by their settings.
Technically, this would mean Apple exposes a function such as:
Label([ManagedSettings.Token])
Which would render a string representation of all the entities included in the array, in a human-
readable, localized manner, taking into conservation the width of the label, abbreviating when
space is limited (e.g. Instagram & 1 more).

## Other Frameworks Evaluated

Yes. We attempted to build our own string composition layer by querying individual labels
(Label(applicationToken)) and concatenating them manually.
However, due to the privacy focus of the API this leads to layout issues (these screen time
token labels always take full width). On top of that, we can’t set / extract the font styling, so it’s
hard to come up with a consistent user-facing rendering.
Our current workaround is to provide vague or less user-friendly summaries like “2 apps and 1
category selected,” which results in a worse user experience than Apple’s own Screen Time
interfaces. For that, we just count the amount of tokens of each entity type, instead of
exposing the real names of the entities.
Therefore, a proper system API would be preferable to ensure parity with Apple’s own Screen
Time experience and consistency across locales.
