## Descriptive Title

Access to a system-level deferred deep link feature through the marketplace installation
process

## Request Type

Interoperability Request

## Feature Name

iOS System-Level Deferred Deep Link and Installation Context Preservation Feature

## Reason for Request

Onside is an alternative app marketplace for iOS operating within the EU under DMA
obligations.
iOS provides a system-level feature called Universal Links: when a user taps a link, iOS can
open the corresponding installed app directly and pass the full link to the app. This is how the
App Store receives deep links — because the App Store is preinstalled on all devices, tapping
a link to a specific app immediately opens the App Store on that app's page. The App Store
receives the link, resolves it, and shows the correct content. Developers can use this to drive
users directly to their app pages and track which links and campaigns led to installs via App
Store Connect.
The App Store’s ability to resolve deep links is not solely due to Universal Links, but due to its
system-level integration and preinstallation, which ensures it can receive and process link
context immediately.
Alternative marketplaces are not preinstalled. When a user taps a link that should lead to an
app in Onside, the flow is fundamentally different: the link opens Onside's website (since the
store app is not yet installed). The user taps "install store." The system installation flow starts.
Once the user confirms the store installation and opens the store app, there is no system-level
mechanism to pass the original link or any parameters from the web session to the store app
session. The link context is lost during the installation process.
This means two things are broken:
1. User experience: A user taps a link to app X on Onside's website, installs the store, opens
the store — and lands on the home screen with no way to be taken to app X's page. The
seamless flow from "tap link → see app page" that the App Store provides is impossible.
2. Developer experience: A developer uses different traffic sources to promote app X. All paths
lead to the store installation page. Once the store is installed, there is no way to connect the
user's session to the original link, traffic source, or campaign — making attribution impossible.
Denying the ability to preserve link context through the installation process effectively breaks a
core operating system feature for alternative marketplaces, resulting in degraded functionality
that does not meet the standard of effective interoperability under Article 6(7) DMA.
We require access to a system-level functionality that preserves the original link and its
parameters through the alternative marketplace installation process — from the web session to
the store app session. This would allow Onside to: (1) show the correct app page immediately
after store installation, providing a seamless user journey equivalent to the App Store; and (2)
pass link parameters to developers so they can attribute installs to specific sources and
campaigns, equivalent to what App Store Connect provides. Without this access, the user
journey is broken and developer attribution is impossible for apps distributed through
alternative marketplaces.

## Product Description

Onside — an alternative app marketplace for iOS that enables developers to distribute their
apps outside the App Store within the EU

## How the Product Will Use the Feature

This feature is specifically required for the first-install scenario, where the store is not yet
present on the device and cannot receive the link through existing Universal Links behavior.
Onside will use the system-level deferred deep link feature to:
1. Seamless app page navigation after store installation: When a user taps a link to app X on
Onside's website and installs the store, the system will preserve the original link. Once the
store app opens for the first time, Onside will retrieve the link and immediately navigate the
user to app X's page — instead of the store home screen.
2. Developer attribution and campaign tracking: The preserved link parameters will allow
Onside to attribute each install to its original traffic source, campaign, or referrer. This data will
be available to developers — equivalent to the attribution and campaign reporting that App
Store Connect provides for App Store installs.
3. Consistent deep link behavior for installed stores: When the store is already installed and a
user taps a link, Universal Links will work as expected — the store opens directly to the correct
app page. The deferred mechanism is needed specifically for the first-install scenario where
the store must be installed before the link can be resolved.

## Where the Product Is/Will Be Offered

European Union

## Other Frameworks Evaluated

We have evaluated the following alternatives:
1. Third-party deep link attribution SDKs (Branch, AppsFlyer, Adjust, etc.): These are paid
solutions that use a probabilistic approach to deferred deep linking — matching users based
on device fingerprinting, IP addresses, and other collected data. This approach is not robust: it
is inherently unreliable, requires collecting significant user data and sharing it with third-party
providers, and does not provide deterministic link resolution. It cannot match the reliability of a
system-level feature.
2. Building a proprietary probabilistic solution: Onside could build its own deferred deep link
matching. However, any solution built outside the OS level can only be probabilistic —
requiring collection of user data (IP address, device characteristics, browser metadata) to
attempt matching between web sessions and app sessions. This raises significant privacy
concerns and still cannot guarantee reliable link delivery.
3. Asking the user to find the app through the store catalog: After installing the store, the user
could manually search for app X in the Onside catalog. This completely breaks the user
experience: the user tapped a direct link to app X, went through the store installation process,
and is now expected to manually search for the app they already identified. This is not a viable
user journey.
None of these alternatives provide reliable, deterministic link passing through the installation
process. Only a system-level feature that preserves link context from the web session to the
store app session can provide interoperability.
Any solution implemented outside the operating system cannot provide deterministic delivery
of link context across the installation boundary, as this boundary is controlled exclusively by
iOS.

## Additional Information

Privacy Safeguards:
- The deferred deep link feature does not expose user identity. Onside only receives the link
URL that the user explicitly clicked in a public context (email, web page, Messages, etc.).
- Onside does not receive information about which user clicked the link or when the link was
clicked, only that a link was requested.
- All link handling remains transparent to users: users see the link source (email, web page,
etc.) and explicitly click it. They are not tracked or attributed without their knowledge.
- Onside receives only the URL explicitly activated by the user, without any additional user or
device-level identifiers.
- The feature respects user choice: users can opt out of specific Onside app installations at any
point.
Technical Details:
- The feature should preserve the original URL (or specific parameters) from the web session
and make it available to the store app after installation is complete.
- Implementation could use the existing MarketplaceKit installation flow — the system already
manages the store installation process and could attach link context to it.
- The store app should be able to retrieve the preserved link on first launch via a system API.
Success Criteria:
- After a user installs Onside from a web page, the store app can retrieve the original URL or
parameters from the web session on first launch.
- Users who tapped a link to a specific app are navigated to that app's page immediately after
store installation, without manual search.
- Developers can attribute installs to specific traffic sources and campaigns using the
preserved link parameters.

May 4, 2026, from Apple to Developer:
We have reviewed your submission and determined that your request falls outside the scope of
Article 6(7) DMA. Specifically, your request asks Apple to create a new functionality on iOS that
would preserve link context and its parameters through the installation process of an
alternative marketplace and which is likewise not available in the context of installation of the
App Store on iOS where it was not installed. For this reason, we are closing your request and
not moving it to the next phase of our interoperability process.
As this determination is based on a non-technical consideration (i.e., whether Apple is legally
obligated to create new features within the meaning of Article 6(7) DMA), it is not eligible for the
dispute resolution mechanism.
This concludes our review under Article 6(7) DMA.
